# llm-d v0.6.0 → v0.7.0 Benchmark Findings

**Hardware:** 2× NVIDIA L40S (48 GB total VRAM), TP=2, SNO on IBM Cloud  
**Models:** Qwen3-0.6B, Qwen3-8B, Qwen3-14B, Qwen3-32B-AWQ  
**vLLM:** 0.17.1 (v0.6.0) → 0.19.1 (v0.7.0)

---

## Slide 1: Baseline shift — vLLM 0.19.1 raises the floor

Qwen3-8B no-offload throughput increased 34% (197→264 tok/s) — the largest
single-version jump in this evaluation series. Independent of KV offload.

| Model | v0.6.0 | v0.7.0 | Δ |
|-------|-------:|-------:|--:|
| Qwen3-0.6B no-offload | 807 tok/s | 811 tok/s | +0.4% |
| **Qwen3-8B no-offload** | **197 tok/s** | **264 tok/s** | **+34%** |
| Qwen3-14B no-offload | 55.5 tok/s | 57.6 tok/s | +3.8% |
| Qwen3-32B-AWQ no-offload | 50.1 tok/s | 50.1 tok/s | 0% |

## Slide 2: Memory-pressure — offload delivers throughput AND latency wins

Under constrained GPU KV-cache (reduced gmu), native-offload cache hits eliminate
re-prefilling the 10,000-token shared prefix — producing throughput gains and TTFT
reduction simultaneously. All percentages compare each offload configuration to
no-offload at the same reduced `gpu_memory_utilization`, within v0.7.0.
Filesystem offload rows are first-ever valid baselines (v0.6.0 was misconfigured).

| Model/Config | Throughput vs mempress no-offload | TTFT vs mempress no-offload |
|---|:---:|:---:|
| Qwen3-0.6B native-offload | **+53%** | **−47%** (3.29 s → 1.75 s) |
| Qwen3-0.6B cpu+fs-offload *(first baseline)* | **+35%** | −38% (2.03 s) |
| Qwen3-14B native-offload | +4.6% | −0.7% TTFT, −6% ITL |
| Qwen3-14B cpu+fs-offload *(first baseline)* | +3.2% | −3.9% ITL |

## Slide 3: Regression — Qwen3-8B OffloadingConnector (vLLM bug, unfixed in main)

Both rows are **Qwen3-8B at gmu=0.9** — the default, unconstrained setting (same as
slide 1). At gmu=0.9 the GPU KV-cache is not full, so the offload connector has no
blocks to evict and no cache hits to serve. It is configured but should be nearly
invisible. Each row compares native-offload throughput against no-offload within the
same version.

| Version | vLLM | native-offload vs no-offload | TTFT at concurrency=50 |
|---------|------|:----------------------------:|:----------------------:|
| v0.6.0 | 0.17.1 | −6.5% | 0.90 s |
| **v0.7.0** | **0.19.1** | **−39.9%** | **11.46 s (12× worse)** |

In v0.6.0, configuring the connector costs just −6.5% — close to zero, as expected.
In v0.7.0 the same configuration causes −39.9% throughput loss and 12× TTFT
inflation, wiping out the +34% baseline gain from slide 1. The connector actively
harms performance despite doing no useful work.

### Root cause: `_blocks_being_loaded` request convoy

vLLM PR #29087 introduced `_blocks_being_loaded` tracking to prevent duplicate
CPU→GPU loads. When 50 concurrent requests share a 10,000-token prefix, the first
request to touch shared blocks places their keys in `_blocks_being_loaded`, and every
other request is delayed (`return None`) until that load completes — a request
convoy. The GPU cache has room, but all requests are serialised anyway.

```python
if any(key in self._blocks_being_loaded for key in offload_keys):
    return None   # ← serialises all concurrent requests through one load
```

### Proposed fix (one-line change)

Return `0` instead of `None`: skip the CPU hit for the blocked request and let it
re-prefill. The first request's load will warm the GPU cache for subsequent requests
without stalling everyone behind a lock.

```python
# scheduler.py ~line 507
if any(key in self._blocks_being_loaded for key in offload_keys):
-   return None   # delay entire request (causes convoy)
+   return 0      # skip CPU hit, proceed without offload for this turn
```

This preserves the "no duplicate loads" intent (only one load job is created) while
eliminating the request convoy.

## Slide 4: Wins — 32B-AWQ unblocked + first valid filesystem offload

### Qwen3-32B-AWQ offload now works (vLLM #38515 resolved)

All 32B-AWQ offload configs previously crashed with a negative Prometheus counter
when block preemptions occurred under KV offload pressure with quantized models
(vLLM #38515). Fixed in vLLM PR #34805 (block-level preemption handling).
In v0.7.0: 0% overhead across all offload configs, all concurrency levels complete.

### First valid filesystem offload

v0.5.1 and v0.6.0 fs-offload results were invalid (system PVC misconfiguration).
v0.7.0 is the first valid baseline with `llmd_fs_connector` v0.19 baked into
the image.

At gmu=0.9: Qwen3-14B −3.5%, Qwen3-32B-AWQ −2.0% overhead vs no-offload.  
Under mempress: Qwen3-0.6B cpu+fs-offload **+34.6%** vs mempress no-offload.

`analysis/v0.7.0_fs_offload_mempress_baseline.png`
