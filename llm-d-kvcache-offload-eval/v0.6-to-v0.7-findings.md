# llm-d v0.6.0 → v0.7.0 Benchmark Findings

**Hardware:** 2× NVIDIA L40S (48 GB total VRAM), TP=2, SNO on IBM Cloud  
**Models:** Qwen3-0.6B, Qwen3-8B, Qwen3-14B, Qwen3-32B-AWQ  
**vLLM:** 0.17.1 (v0.6.0) → 0.19.1 (v0.7.0)

---

## Slide 1: Baseline shift — vLLM 0.19.1 raises the floor

Qwen3-8B no-offload throughput increased 34% (197→264 tok/s) — the largest
single-version jump in this evaluation series. Independent of KV offload.

| Model | v0.6.0 | v0.7.0 | Δ |
|-------|-------:|-------:|--:|
| Qwen3-0.6B no-offload | 807 tok/s | 811 tok/s | +0.4% |
| **Qwen3-8B no-offload** | **197 tok/s** | **264 tok/s** | **+34%** |
| Qwen3-14B no-offload | 55.5 tok/s | 57.6 tok/s | +3.8% |
| Qwen3-32B-AWQ no-offload | 50.1 tok/s | 50.1 tok/s | 0% |

**Graph:** `analysis/v0.7.0_version_comparison.png`

## Slide 2: Memory-pressure — offload delivers throughput AND latency wins

Under constrained GPU KV-cache (reduced gmu), native-offload cache hits eliminate
re-prefilling the 10,000-token shared prefix. This produces throughput gains AND
TTFT reduction simultaneously.

| Model/Config | Throughput vs no-offload | TTFT vs no-offload |
|---|:---:|:---:|
| Qwen3-0.6B native-offload (mempress) | **+53%** | **−47%** (3.29 s → 1.75 s) |
| Qwen3-0.6B cpu+fs-offload (mempress) | **+35%** | −38% (2.03 s) |
| Qwen3-14B native-offload (mempress) | +4.6% | −0.7% TTFT, −6% ITL |
| Qwen3-8B native-offload (mempress) | −18% | +14% TTFT |

Qwen3-8B remains negative — the 6,000 IOPS PVC write rate dominates.

`analysis/v0.7.0_mempress_ttft_curves.png`

## Slide 3: Regression — Qwen3-8B OffloadingConnector (vLLM bug, unfixed in main)

### Measured impact (gmu=0.9, GPU KV-cache NOT constrained)

| Version | vLLM | Throughput vs no-offload | TTFT at rate=50 |
|---------|------|:------------------------:|:---------------:|
| v0.5.1 | 0.15.1 | −29.9% | ~0.9 s |
| v0.6.0 | 0.17.1 | −6.5% | ~0.9 s |
| **v0.7.0** | **0.19.1** | **−39.9%** | **11.46 s (12× worse)** |

### Root cause: `_blocks_being_loaded` head-of-line blocking

Commit `7013e9ac8` (PR #29087, "Prevent redundant loads", Jan 21 2026) introduced
tracking of in-flight CPU→GPU load jobs:

```python
# scheduler.py line 293 (HEAD 8a9eb408, confirmed unfixed)
self._blocks_being_loaded: set[OffloadKey] | None = (
    set() if spec.vllm_config.cache_config.enable_prefix_caching else None
)
```

When request A triggers a load of shared prefix blocks, ALL requests sharing that
prefix are delayed (line 507, `return None`) until A's load completes:

```python
if any(key in self._blocks_being_loaded for key in offload_keys):
    return None   # ← serialises all concurrent requests through one load
```

With 50 concurrent requests sharing a 10,000-token prefix, the first request to
trigger an offload load serialises the other 49 behind it. This explains both the
throughput collapse and the 12× TTFT increase — it is head-of-line blocking, not
load latency.

### Proposed fix (one-line change)

Return `0` instead of `None`: skip the CPU hit for the blocked request and let it
re-prefill. The first request's load will warm the GPU cache for subsequent requests
without stalling everyone behind a lock.

```python
# scheduler.py ~line 507
if any(key in self._blocks_being_loaded for key in offload_keys):
-   return None   # delay entire request (causes convoy)
+   return 0      # skip CPU hit, proceed without offload for this turn
```

This preserves the "no duplicate loads" intent (only one load job is created) while
eliminating the request convoy.

### Status

- **Introduced:** vLLM PR #29087 / commit `7013e9ac8` (between v0.17.1 and v0.19.1)
- **Latest main:** Still present — verified against `HEAD 8a9eb408` (2026-05-25)
- **Issue:** [vllm-project/vllm#44294](https://github.com/vllm-project/vllm/issues/44294)
- **Candidate fix:** [vllm-project/vllm#44295](https://github.com/vllm-project/vllm/pull/44295)

## Slide 4: Wins — 32B-AWQ unblocked + first valid filesystem offload

### Qwen3-32B-AWQ offload works (vLLM #38515 resolved)

All 32B-AWQ offload configs previously crashed (negative Prometheus counter under
high preemption). Now: 0% overhead across all offload configs.

Fixed by commit `fcf0687b2` (PR #34805, "Support block-level preemption handling")
which addressed atomic reference counting for quantized models.

### First valid filesystem offload

v0.5.1 and v0.6.0 fs-offload results were invalid (system PVC misconfiguration).
v0.7.0 is the first valid baseline with `llmd_fs_connector` v0.19 baked in.

At gmu=0.9: Qwen3-14B −3.5%, Qwen3-32B-AWQ −2.0% overhead.  
Under mempress: Qwen3-0.6B cpu+fs-offload **+34.6%** vs mempress no-offload.

**Graphs:** `analysis/v0.7.0_overhead_heatmap.png`,
`analysis/v0.7.0_fs_offload_mempress_baseline.png`
